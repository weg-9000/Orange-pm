## Division of labor with auto-generation (journey_build.py)

The standard storyboard (screen order · draft status · transition skeleton) is **auto-generated by
`scripts/journey_build.py` via a PostToolUse hook each time a draft is edited**, refreshing
`reports/journey-latest.md` (0 LLM tokens, not recorded in session-log). The user journey board
in the viz prototype view always displays this file (journey_emit picks the most recent mtime).

This skill handles the **judgment areas** the auto-builder cannot:
- `--actor` filtering, `--from` partial journey
- Narrative enrichment of key actions · transition conditions · drop-off paths (the auto version
  uses the `[auto-generated — enrich with /journey]` placeholder)
- Saving timestamped versions (`journey-{YYYYMMDD-HHMM}.md`) + session-log recording


## Authoring model branch (dossier vs section/screen)

This skill supports two models. The model is determined before the precondition check.

| Model | Determination condition | Screen source |
|---|---|---|
| **dossier (Track A)** | `work-orders/cluster_index.json` exists **or** a `SUPERSEDED` banner exists in `graph/screen-list.md` | Screen items in the **`§2 Screens`(or `## §2`) section** of each dossier draft (`cluster_{cluster_id}.draft.md`) pointed to by `clusters[].draft_path` in `cluster_index.json`. **Do not hardcode the filename — follow `draft_path`** (2026-06-08 H6 audit finding) |
| **section/screen (legacy)** | The above condition is not met but SCR rows exist in `screen-list.md` | `screen-list.md` + screen WO `drafts/{WO_ID}.draft.md` |

In the dossier model, screens aren't separate screen WOs/nodes — they're contained in the §2
Screens section of each capability dossier (requirements.md FR delegation principle). Below,
the phrase "screen (SCR-NNN)" should be read as "dossier §2 screen item" in the dossier model.


## Precondition check

1. After **determining the model**, confirm the screen source.
   - dossier model: verify `work-orders/cluster_index.json` and the §2 Screens section of each
     dossier exist. If a screen section is empty, mark that dossier `[§2 not written]` and
     continue.
   - section/screen model: verify `graph/screen-list.md` exists. If not, guide the PM to
     `/se` or `/graph-gen` and stop.
   - If neither exists, guide the PM to `/graph-gen {product}` (graph generation) or dossier
     authoring, and stop.

2. Check whether `PROJECTS/{product}/inputs/requirements.md` exists.
   If not, use source order (dossier: cluster order / legacy: screen-list row order) as-is,
   without FR-order-based sorting.

3. If the `--from {screen_id}` option is given, verify the Screen ID exists in screen-list.md.
   If not, output the list of valid Screen IDs and stop.

4. If the `--actor {actor}` option is given, cross-check against the Layer 4 actor list in
   requirements.md. For an unlisted actor, output a warning and confirm with the PM whether to
   proceed.

This skill is read-only. It does not modify any file (except saving to reports/).


## Execution steps

### Step 1 — Reconstruct screen order

- **dossier model**: read the §2 Screens section of each dossier draft
  (`clusters[].draft_path`, typically `cluster_{cluster_id}.draft.md`) in cluster order
  (capability) from `cluster_index.json`, and extract/list the screen items. If no screen ID
  exists, assign one as `{cluster_id}-S{n}`.
- **section/screen model**: read all rows of `screen-list.md`.

**Sort order (in priority order):**
1. requirements.md Layer 1 FR order Must → Should → Could (FR ↔ screen: dossier uses the FR
   reference in the §2 item, legacy uses the screen-list REQ column)
2. If FR order is identical, keep source order (dossier: cluster order / legacy: screen-list
   row order)

When `--from {screen_id}` is given: start from that screen and summarize prior screens in a
"preceding screens" section.
When `--actor {actor}` is given: include only screens with a REQ-NNN associated with that actor.

Output the number of screens to process.


### Step 2 — Collect draft status

Check the status of each screen.

- **dossier model**: screen status is determined by the `review_status` (canonical) in the
  frontmatter of the owning dossier draft (`clusters[].draft_path`). `human-reviewed` (or
  legacy-compatible `reviewed:true`) → ✅ · `ai-draft` + §2 Screens present → 📝 · §2 Screens
  not written → ⬜.
- **section/screen model**: determined per-screen by the screen WO draft (table below).

| Status | Condition | Storyboard notation |
|---|---|---|
| Complete | `drafts/{WO_ID}.draft.md` exists + `reviewed: true` (or dossier `status: human-reviewed`) | ✅ |
| In progress | draft exists + `reviewed: false` (or dossier `status: ai-draft`/`draft`) | 📝 |
| Sketch | `sketches/{screen_id}.sketch.md` exists (legacy only) | 🔲 |
| Not started | neither draft nor sketch exists (dossier: §2 Screens not written) | ⬜ |


### Step 3 — Extract inter-screen transition conditions

Read the transition condition for each screen from the following sources in priority order:

| Priority | Source | Extracted item |
|---|---|---|
| 1 | `drafts/{WO_ID}.draft.md` success state | "next action list" |
| 2 | `sketches/{screen_id}.sketch.md` | transition-related narrative |
| 3 | `screen-list.md` purpose column | screen purpose text |
| 4 | none | `[transition condition undetermined]` |

Extract drop-off points from each screen's error state or cancel/back-navigation flow.


### Step 4 — Output storyboard

```
Customer Journey Storyboard — {product}
Actor: {actor or "all"} / Generated at: {UTC}
Total {N} screens (Complete {N} ✅ / In progress {N} 📝 / Sketch {N} 🔲 / Not started {N} ⬜)
─────────────────────────────────────────────────────────────────
```

Output each screen in the following format:

```
[{seq}] {SCR-NNN} {screen name}  {status icon}
  Entry condition: {idle-state entry condition or previous screen's transition condition}
  Key action:       {the most important action the customer takes on this screen}
  Transition:       → {next screen SCR-NNN} ({transition condition})
  Drop-off:         ✕ {drop-off path} ({drop-off condition})  ← only if present
```

Not-started/sketch screens are shown in the following format:
```
[{seq}] {SCR-NNN} {screen name}  ⬜ Not started
  Purpose: {screen-list.md purpose column}
  REQ:     {REQ-NNN}
  Transition: [undetermined]
```

Output a summary section at the end of the full journey:
```
─────────────────────────────────────────────────────────────────
Journey summary
  Entry point:        {first screen}
  Key path:            {SCR-001} → {SCR-002} → ... → {last screen}
  Major drop-offs:     {list of drop-off points}
  Undetermined stretch: {list of screens with undetermined transition conditions}

Screens in progress:
  📝 {WO_ID} {screen name} — recommend /review drafts/{WO_ID}.draft.md
  ⬜ {WO_ID} {screen name} — (dossier model) write §2 Screens via /write-cluster / (legacy) /flow {product} {SCR-NNN}
```


### Step 5 — Save file

Save to `PROJECTS/{product}/reports/journey-{YYYYMMDD-HHMM}.md`.

File header:
```markdown
---
generated_at: {UTC timestamp}
product: {product}
actor: {actor or all}
from_screen: {screen_id or first}
screen_count: {N}
draft_complete: {N}
draft_in_progress: {N}
sketch_only: {N}
not_started: {N}
---
```


### Step 6 — Record in session-log

```markdown
- {date} /journey: generated storyboard for {N} screens / complete {N} / not started {N}
```

## Usage examples

```bash

/journey dbaas

# Check starting from a specific screen
/journey dbaas --from SCR-003

# Filter by a specific actor's perspective
/journey dbaas --actor "Service Manager"

# Mid-progress check during Phase 2 work
/journey dbaas
```


## Notes

- This skill **never modifies content** — it is a read-only synthesized view.
- Callable at any time (no Phase restriction, same tier as `/render`).
- Does not stop even if there are unfinished draft screens.
- The storyboard content is a **progress-check tool**, not a deliverable.
  The source of truth is always each draft file and screen-list.md.
